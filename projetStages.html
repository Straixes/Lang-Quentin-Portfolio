Dossier 03 / SAE 4.REAL.01

Mon carnet de suivi de stage

Amélioration et optimisation d'une application existante de suivi de recherche de stage, composée d'une application mobile Android, d'un back-office web Symfony et d'une base de données PostgreSQL. Mon travail s'est concentré sur l'application mobile et la stratégie de tests de bout en bout.

Aperçu Équipe Travaux Technique Bilan
Le projet en bref

Vue d'ensemble

SAE 4.REAL.01 — Développement d'une application complexe, S4 BUT Informatique.

Objectif

Évaluer la qualité d'une application existante — « Mon carnet de suivi de recherche de stage » — puis proposer et mettre en œuvre des optimisations sur la conception, l'ergonomie, la performance et la fiabilité.

Architecture

Écosystème en trois parties reliées par une API : une application mobile Android (Java, Retrofit), un back-office web (Symfony / API Platform, MVC, Twig) et une base de données PostgreSQL exploitée via Doctrine.

Équipe

Groupe de 6 étudiants, divisé en deux sous-groupes de 3 : un sur le back-office web, l'autre sur l'application mobile. Synchronisation régulière en fin de séance pour garder les solutions cohérentes.

Qui a fait quoi

Organisation de l'équipe

Deux sous-groupes de trois, répartis selon les compétences de chacun.

Sous-groupe Back-office Web

Restructuration de l'architecture Symfony (contrôleurs, services, repositories, entités), refonte des interfaces (tableau de bord, suivi des offres) et mise en conformité avec les standards d'accessibilité.

  • Conteneurisation Docker
  • Refonte du Tableau de Bord (graphiques)
  • Services métier & entités Symfony
  • Tests PHPUnit (sécurité, contrôleurs, services)

Sous-groupe Mobile & Tests (le mien)

Refonte de l'expérience de l'application Android destinée aux étudiants, et mise en place d'une stratégie de tests d'interface de bout en bout pour fiabiliser les parcours critiques de l'application.

  • Refonte visuelle & navigation Android
  • Indicateurs visuels & feedback (Toasts)
  • Tests End-to-End sur les parcours clés
  • Diagnostic des instabilités réseau via les tests

Mon rôle dans le sous-groupe

Au sein du sous-groupe Mobile, je me suis concentré sur l'application Android destinée aux étudiants et sur la mise en place des tests d'interface (End-to-End) permettant de valider les parcours utilisateurs les plus importants.

  • Responsable principal des tests front-end et End-to-End : définition de la stratégie, rédaction et exécution des scénarios de test sur les parcours critiques de l'application mobile.
  • Réalisation des modifications et améliorations sur l'application Android : refonte de la navigation, cohérence visuelle entre les écrans, ajout d'indicateurs de statut et de retours utilisateur (Toasts).
  • Rôle de second chef de projet à l'échelle du groupe entier (6 personnes) : coordination des tâches entre les deux sous-groupes, suivi de l'avancement global et point de liaison pour maintenir la cohérence entre la partie mobile et le back-office web.
Du diagnostic à la correction

Travaux réalisés — Mobile & Tests

La démarche suivie sur la partie sur laquelle j'ai travaillé.

01/04

Rétroconception & diagnostic

Analyse de l'application Android existante : navigation désordonnée, barre d'activité créant des conflits visuels, marges et ordres de boutons incohérents entre les écrans.

02/04

Refonte de l'interface mobile

Généralisation d'une barre de navigation cohérente, repositionnement et renommage des boutons « Retour », ajout de pastilles de couleur indiquant le statut des candidatures et des offres, et messages « Toast » pour confirmer chaque action critique.

03/04

Stratégie de tests End-to-End

Priorisation des tests d'interface plutôt que des tests unitaires isolés, afin de valider la chaîne complète — interface, appels API et persistance des données. Couverture des parcours de navigation et d'authentification, incluant la vérification des messages d'erreur en cas d'identifiants invalides.

04/04

Diagnostic via les tests

Les échecs rencontrés pendant les tests automatisés ont permis d'isoler des failles d'infrastructure : coupures prématurées de flux JSON et problèmes de configuration serveur. Ajout de délais ciblés pour fiabiliser les appels asynchrones face aux latences.

Stack technique

Technologies & outils utilisés

Application mobile

Application Android en Java (architecture MVC), interfaces définies en XML, appels réseau via Retrofit vers l'API du back-office.

Tests End-to-End

Stratégie de tests d'interface priorisée sur les parcours critiques : navigation, authentification et gestion des erreurs réseau.

Back-office Symfony

Framework Symfony / API Platform suivant le modèle MVC, ORM Doctrine, vues Twig — exposant l'API consommée par l'application mobile.

Base de données

PostgreSQL, avec une réflexion de normalisation vers la 3e forme normale (3FN) menée en parallèle par le sous-groupe web.

Conteneurisation

Image Docker basée sur PHP 8.4, avec Composer 2 et les dépendances nécessaires à la connexion PostgreSQL, pour un environnement portable.

Qualité & ergonomie

Audit basé sur les critères de Bastien & Scapin et sur la norme ISO 25000, avec mesure d'acceptabilité via score SUS et protocole « Think Aloud ».

Bilan

Ce que ce projet m'a apporté

Au-delà des aspects techniques, ce projet a aussi été une préparation à l'arrivée en stage.

  • Tests End-to-End sur mobile : première expérience concrète de mise en place d'une stratégie de tests d'interface sur une application Android, couvrant des parcours complets — de l'authentification jusqu'à la validation des données affichées. J'ai compris que les tests peuvent servir autant à diagnostiquer des problèmes d'infrastructure qu'à valider des fonctionnalités.
  • Développement mobile Android : prise en main d'une base de code Android existante en Java — sans tout reconstruire — pour l'améliorer de façon ciblée. Travail sur la navigation, la cohérence visuelle entre écrans et les retours utilisateur, dans un contexte où chaque modification devait rester compatible avec l'API existante.
  • Coordination d'équipe : en endossant le rôle de second chef de projet, j'ai appris à répartir les tâches, suivre l'avancement de chacun et maintenir le lien entre les deux sous-groupes (mobile et web) qui travaillaient sur des parties distinctes mais dépendantes l'une de l'autre via l'API.
